# 2026-04-05 K20 Pro 安装 Gemma 4 部署记录

## 背景

目标是在一台红米 K20 Pro 上，通过原生 Termux 部署 Gemma 4，并验证文本与多模态能力是否能够实际运行。

设备与环境：

- 设备：Redmi K20 Pro
- SoC：Snapdragon 855
- 内存：12 GB RAM
- 系统环境：Android + 原生 Termux
- 远程管理方式：SSH 连接到 `u0_a247@192.168.2.202:8022`

## 目标

- 在手机上完成 Gemma 4 的本地部署
- 能够稳定运行文本推理
- 补齐多模态运行链路
- 记录实际资源占用与限制

## 部署过程

### 1. 基础环境准备

先在 Termux 中准备常用构建与运行环境：

- `termux-wake-lock`
- `clang`
- `cmake`
- `git`
- `wget`

这样做的目的是避免后台休眠影响长时间编译与模型加载。

### 2. 编译 llama.cpp

在手机上拉取并编译 `llama.cpp`，最初构建出了：

- `llama-llava-cli`

后续实际测试中发现，当前版本的 `llama-llava-cli` 已被废弃，只会提示改用：

- `llama-mtmd-cli`

因此又额外补编了：

- `llama-mtmd-cli`

这一步非常关键，因为多模态最终依赖的是 `llama-mtmd-cli`，而不是旧的 `llava` 工具。

### 3. 下载模型

部署时使用的主要文件如下：

- 文本模型：`gemma-4-E4B-it-Q4_K_M.gguf`
- 视觉投影：`mmproj-BF16.gguf`

在手机上的实际文件整理为：

- `~/models/gemma-4-main.gguf`
- `~/models/mmproj-vision.gguf`

文件体量大致为：

- 文本模型约 `5.3 GB`
- 视觉投影约 `946 MB`

### 4. 文本服务方案选择

最开始考虑过直接使用 `llama-server`，但在 Android / Termux 环境下遇到兼容性问题。

主要原因是：

- `llama-server` 依赖的部分实现使用了 `posix_spawn`
- Android 的 Bionic libc 对这一点不兼容

因此最终选择改用：

- `Ollama 0.20.0`

在 Termux 中直接通过包管理安装 `ollama`，作为文本推理服务。

### 5. 文本推理验证

文本侧验证方式比较直接：

```bash
ollama run gemma4 "ping"
```

返回结果正常，说明文本推理链路可用。

### 6. 多模态链路排查

多模态部分一开始并不是一次成功，主要经历了几轮排查：

1. 旧版 `llama-llava-cli` 只能输出废弃提示，无法真正执行推理
2. 重新编译 `llama-mtmd-cli`
3. 首次运行时，Gemma 4 模板要求加 `--jinja`
4. 再次运行后，多模态链路正式打通

实际测试图像使用的是 `llama.cpp` 自带图片：

- `~/llama.cpp/media/llama1-logo.png`

多模态测试命令思路如下：

```bash
~/llama.cpp/build/bin/llama-mtmd-cli \
  --jinja \
  --no-warmup \
  -m ~/models/gemma-4-main.gguf \
  --mmproj ~/models/mmproj-vision.gguf \
  --image ~/llama.cpp/media/llama1-logo.png \
  -p "请用中文简短描述这张图片，并判断它更像 logo、横幅还是图标。" \
  -c 2048 \
  -n 80 \
  -t 8
```

测试结果表明：

- 图像编码成功
- 图像 token 解码成功
- 文本生成成功

虽然输出中带有 `thought` 通道，格式不够干净，但模型已经能够识别图片内容，并判断这张图更偏向 `logo`。

## 关键结论

### 文本能力

Gemma 4 在 K20 Pro 上可以运行文本推理，且已经通过 Ollama 完成验证。

### 多模态能力

多模态可以跑通，但条件是：

- 使用 `llama-mtmd-cli`
- 使用 `--jinja`
- 搭配正确的 `mmproj` 文件

### 当前限制

虽然能跑通，但速度并不轻松，属于“可用但不轻盈”的状态。

## 实测资源占用

多模态测试时观测到的大致情况如下：

- `llama-mtmd-cli` 进程 CPU 占用大约在 `692%` 到 `720%`
- 基本接近吃满 8 核
- 进程 RSS 高峰大约到 `7.1 GB`
- 常驻运行阶段大约在 `6.8 GB` 左右
- `MemAvailable` 最低下降到约 `5.18 GB`
- swap 也开始参与使用

推理时间大致为：

- 图像编码约 `92.7s`
- 图像解码约 `18.7s`
- 总耗时约 `143.4s`

这说明：

- K20 Pro 的 12 GB 内存足以让这套组合跑起来
- 但如果模型再继续明显增大，速度和发热都会更吃紧

## 关于模型大小的判断

虽然文件名里写的是 `E4B`，但在 `llama.cpp` 读取出的元数据中，模型参数量显示约为：

- `7.52B`

也就是说，这次跑起来的并不是一个特别小的纯 4B 级别模型。

因此从实测来看：

- 当前这档已经接近手机 CPU 推理的较高负载区间
- 更大模型理论上仍可尝试
- 但收益会很快被速度、发热和 swap 消耗抵消

## 运行方式参考

### 文本

```bash
ollama serve > ~/ollama.log 2>&1 &
ollama run gemma4
```

### 多模态

```bash
~/llama.cpp/build/bin/llama-mtmd-cli \
  --jinja \
  --no-warmup \
  -m ~/models/gemma-4-main.gguf \
  --mmproj ~/models/mmproj-vision.gguf \
  --image /path/to/your-image.jpg \
  -p "请描述这张图片" \
  -c 2048 \
  -n 80 \
  -t 8
```

## 遇到的问题

这次部署过程中，比较关键的问题有这些：

- `llama-server` 在 Android / Termux 环境下兼容性不理想
- `llama-llava-cli` 已经废弃
- Gemma 4 多模态模板需要 `--jinja`
- 手机相册目录权限在 Termux 下需要额外处理，不能默认直接读取

## 最终结论

这次部署是成功的。

对于 K20 Pro 这类老旗舰设备来说：

- 跑 Gemma 4 文本推理是可行的
- 跑 Gemma 4 多模态也是可行的
- 但整体更适合作为“可验证、可使用、可折腾”的移动本地 AI 方案，而不是高吞吐或低延迟方案
